Game Files Reference

Neodyme serves several configuration files to Fortnite clients via the cloud storage system. Some are static, some are dynamically generated per client version. All can be edited from the Dev Panel -> Config Files in the dashboard.

On this page

Overview

When Fortnite starts, it fetches files from GET /fortnite/api/cloudstorage/system. These files override the game's built-in settings. Neodyme generates some files at runtime based on server.properties and config JSON files.

FileLocationTypeGenerated
DefaultEngine.ini data/cloudstorage/system/ INI On startup - XMPP host/port, regions
DefaultGame.ini data/cloudstorage/system/ INI Per request - playlist section rebuilt from playlists.json
DefaultRuntimeOptions.ini data/cloudstorage/system/ INI Per request - bForceBRMode, bSkipSubgameSelect injected from build 19.30+
playlists.json config/ JSON Static - source of truth for game modes
shop.json config/ JSON Static - item rotation and shop categories
experience.json config/ JSON Static - XP tables and level thresholds
game-servers.json config/ JSON Static - matchmaking regions and server IPs
motd.json config/ JSON Static - Message of the Day shown in lobby

config/playlists.json JSON

Single source of truth for which game modes appear in the Fortnite lobby. Used by two systems:

  • Cloud Storage - DefaultGame.ini FrontEndPlaylistData section is rebuilt from this file for each client request, filtered by the client's build number.
  • Discovery Surface - POST /api/v2/discovery/surface/* results are filtered by linkCode and minBuild.

Fields

FieldTypeDescription
namestringUE4 playlist name (e.g. Playlist_Trios)
linkCodestringDiscovery surface identifier (e.g. playlist_trios, campaign)
enabledbooleanMaster switch - false hides regardless of build
minBuildnumberMinimum Fortnite build required (0 = always shown). e.g. 15.0, 19.30
ini.bIsDefaultPlaylistbooleanMarks as the default selected mode
ini.bVisibleWhenDisabledbooleanShow grayed-out tile when bEnabled=false
ini.bDisplayAsNewbooleanShow "NEW" badge on the tile
ini.bInvisibleWhenEnabledbooleanHide tile even when enabled (for background playlists)
ini.categoryIndexnumber0 = main tab, 1 = LTM tab
ini.displayPrioritynumberOrder within the category (lower = first)
Tip: Changes to playlists.json take effect immediately - no restart needed. The file is read on each cloud storage request.

DefaultGame.ini INI

data/cloudstorage/system/DefaultGame.ini

Controls gameplay settings, chat configuration, text replacements, and playlist visibility. The FrontEndPlaylistData block is replaced dynamically at request time - edits to that section in the file will be overwritten. All other sections are served as-is.

Key sections

SectionNotable keys
[/Script/FortniteGame.FortTextHotfixConfig] Text replacements shown in-game (e.g. "Logging Into Neodyme...")
[/Script/FortniteGame.FortChatManager] Global/founder/Athena chat toggles, reconnect delays
[/Script/FortniteGame.FortGameInstance] FrontEndPlaylistData - auto-generated from playlists.json
Note: Do not manually edit the FrontEndPlaylistData lines - they are overwritten on every request. Use config/playlists.json instead.

DefaultEngine.ini INI Auto-generated

data/cloudstorage/system/DefaultEngine.ini

Generated on every server startup from server.properties. Do not edit manually - changes are overwritten on restart.

Key settings (from server.properties)

server.properties keyINI fieldDescription
xmppHostServerAddrXMPP WebSocket address sent to clients. Set to your LAN/public IP for multi-machine setups.
xmppPortServerPortXMPP WebSocket port (default: 80)
Multi-machine setup: Set xmppHost=192.168.x.x (your server's LAN IP) in server.properties so clients on other machines can connect to XMPP. Restart the server to regenerate DefaultEngine.ini.

DefaultRuntimeOptions.ini INI

data/cloudstorage/system/DefaultRuntimeOptions.ini

Runtime feature flags. Most fields are static, but two are injected dynamically depending on the client's build:

KeyDefaultDynamic behaviour

bSkipSubgameSelect false Set to true for builds ≥ 19.30 - hides the STW/BR/Creative selection screen
bDisableGifting false Enable/disable item gifting globally
bEnableGlobalChat true Enable global chat channel in lobby
DaysOfFriendshipBeforeCanGift 3 Days two players must be friends before gifting is allowed
bSkipTrailerMovie false Skip the intro video on launch

Other config/ JSON files

config/shop.json

Defines shop categories, item rotation rules, and which items are eligible for the daily/featured rotation. See the Configuration page for the full reference.

config/experience.json

XP requirements per level and season XP multipliers.

config/game-servers.json

Matchmaking regions and dedicated game server addresses. Changes here are reflected in the next DefaultEngine.ini regeneration (server restart). See the Matchmaking section below for the full configuration walkthrough.

config/motd.json

Message of the Day shown in the lobby news panel. Supports multiple entries with title, body, image, and tab name.

Asset Pipeline (images & CDN) 1.2.6+

Fortnite hard-codes asset URLs like https://fortnite-public-service-prod11.ol.epicgames.com/images/<path>. These requests hit Neodyme thanks to a DNS / hosts rewrite. The Asset Pipeline decides, per request, whether to serve the file from disk or redirect to the official Epic CDN.

Two modes

ModeBehaviorDisk usageNetwork
online default Redirect (HTTP 302) to the official Epic CDN listed in content/assets-index.json None (beyond alwaysLocal) Client fetches from Epic CDN directly. Zero backend bandwidth.
local Serve the file from public/images/<path>. Missing files fall back to the CDN with a warning log. ~36 MB of base assets, more depending on what you ship. None at runtime (after initial install).

First-launch prompt

The first time Neodyme boots without an assetsMode in server.properties, an interactive prompt asks [O]nline / [L]ocal and persists the choice. In non-interactive environments (CI, Docker) the default is online.

# server.properties (section [3b])
assetsMode=online

Key files

FilePurposeEdited by
content/assets-index.json Source of truth mapping public/ paths to their official Epic CDN origin. Used by both modes. Generated via maintenance script; manual edits possible.
public/images/<path> Local copy of every served asset. Always required for alwaysLocal folders. /assets install populates from the remote manifest.
(remote) Neodyme-Plugins/data/assets.json Manifest listing every downloadable asset with its raw.githubusercontent.com URL. Curated on the Neodyme-Plugins repository.

alwaysLocal

Folders matched by patterns under alwaysLocal in content/assets-index.json are never redirected, no matter the mode. They are part of Neodyme branding and ship with the codebase.

"alwaysLocal": [
  "images/neodyme-public-service/**",
  "images/radios/**"
]

How a request resolves

Client GET /images/unrealengine-cdn2/foo.png
       │
       ▼
asset-middleware (before express.static)
       │
       ├─ alwaysLocal? ─── yes ──► serve from public/  (always)
       │
       ├─ mode = local? ──┬─ file present ──► serve from public/
       │                  └─ file missing ──► 302 redirect + warn log
       │
       └─ mode = online? ─┬─ index has CDN ──► 302 redirect (no warn)
                          └─ no CDN ──┬─ local present ─► serve from public/
                                      └─ neither ──► 404
Why both modes can fall back: some assets (your own banners, posters in fortnite-public-service-prod11/) have no Epic CDN equivalent - they exist only on your backend. In online mode, the middleware silently falls back to express.static for those. In local mode, the reverse is true: a missing local file falls back to the CDN.

Console commands

See the full reference on the Commands page. The most useful ones:

/assets statusShow current mode, indexed/missing counts, local disk usage.
/assets mode <online|local>Switch modes. online -> local triggers an automatic download of missing assets.
/assets install [path]Download all missing assets, or a single one. Sizes are validated post-download.
/assets verifyAudit local files vs the index: detect missing, zero-byte, size mismatches, orphans.
/assets diagnose <path>Trace how a specific /images/ request resolves.
Destructive commands (/assets uninstall, /assets clean) print a heads-up and require /confirm assets-uninstall or /confirm assets-clean to actually delete files.

Matchmaking & dedicated servers 1.2.6+

Matchmaking is fully functional and routes players to the dedicated server that matches their region and game mode. There is no in-game UI to set this up - everything happens in config/game-servers.json plus a host account with the server role. The whole file is cached at startup, so changes to config/game-servers.json require a Neodyme restart (or /reload config) to take effect.

How it works

1. Game sends GET /fortnite/api/game/v2/matchmakingservice/ticket/player/<accountId>
   ?bucketId=<buildId>:<unknown>:<region>:<playlist>

2. Neodyme parses bucketId -> { buildId, region, playlist }
   Looks up the playlist in game_modes (by playlist_id, legacy_id, or playlist).
   Resolves the matching gamemode (BR_Solo, BR_Duos, ...).

3. findAvailableServer(gamemode, region) iterates config.regions[region].servers
   Picks the first server with status == "online" and matching gamemode that
   is not currently assigned. Falls back to ANY region if the requested region
   has none available - this is the normal behaviour when the client sends
   region "NONE".

4. The Dedicated Server then queries the joining player's profile through
   POST /fortnite/api/game/v2/profile/<playerAccountId>/dedicated_server/QueryProfile
   using the bearer token of its OWN host account. That host account MUST have
   the "server" role - see Step 0 below.

5. The chosen server.ip and server.port are sent back to the client as the
   session address. The client connects directly to that dedicated server.

Step 0 - create a host account with the server role

The Dedicated Server (RebootDS, custom DS, ...) signs in as a regular Fortnite account and uses its bearer token to query joining players' profiles. That account must have the server role - otherwise POST .../dedicated_server/QueryProfile returns 403 and the DS falls back to random cosmetics (or refuses the player entirely, depending on the DS build).

Register the host account from the website (e.g. Host0001 / host@...), then from the Neodyme console:

/account role Host0001 server         # promote the account to the "server" role
/account info Host0001                # confirm: role = server (clientType = 5)

Available roles (lowest to highest): player, mod/moderator, dev/developer, admin, owner, server. Only server unlocks the dedicated_server/* MCP routes. You can also promote via the Admin Panel → Users → role dropdown.

Keep host-account credentials private and never reuse them as a player account. The server role bypasses the player-ownership check on every MCP route under dedicated_server/*.

Step 1 - declare your dedicated server

Open config/game-servers.json and add one entry under the region you want, in the servers array. The minimum fields needed are:

"EU": {
  "name": "Europe West",
  "code": "EU",
  "servers": [
    {
      "id": "eu-local-01",                 // any unique string
      "name": "[DS]neodyme-eu-local-01",   // display name, free-form
      "ip": "127.0.0.1",                   // where RebootDS / your DS listens
      "port": 7777,                        // your DS UDP port
      "status": "online",                  // must be "online" to be picked
      "gamemode": "BR_Solo",               // must match a key in game_modes
      "region": "EU"
    }
  ]
}

Important: one server entry = one playlist. If you want Solo and Duos available, declare two entries (same IP, different ports) or two separate dedicated server instances.

Step 2 - check the game_modes section

The game_modes object maps your server.gamemode to the playlist Fortnite expects. The default file already ships with BR_Solo, BR_Duos, BR_Squads, Creative and the two Arena_* modes. Each entry must have:

  • playlist - the in-game playlist name (e.g. Playlist_DefaultSolo).
  • playlist_id - the lower-case id used in matchmaking buckets (e.g. playlist_defaultsolo).
  • legacy_id - the numeric or legacy string id the client sometimes sends instead of the playlist_id.
  • time_between_games - minutes before a server is recycled and made available again after a match (default 25).

Step 3 - allowed client build versions

Two different version controls exist, and it is worth knowing what each one actually does today:

  • Backend gate (server.properties): customVersion=true (default) accepts any Fortnite version that contacts the backend; customVersion=false + fnVersion=<X.YY> rejects everything else (and derives the shop era from fnVersion). This is the only check that currently filters by version.
  • Per-server version field in config/game-servers.json: present on every entry (e.g. "version": "20.00"), but currently not honoured by the matchmaker. findAvailableServer only matches on gamemode + status, so a 15.30 client and a 20.00 client both land on the same server if the playlist matches. This is on the roadmap (planned for 1.2.8) - until then, fill the field for documentation only and rely on the backend gate, or run a separate Neodyme instance per build.

Step 4 - matchmaking_settings

Tuning knobs at the bottom of config/game-servers.json. The two most useful for local testing:

  • min_players_to_start - default 2. The flow waits until that many players are queued before assigning a server. Set to 1 for solo testing. Requires a restart - the value is cached at startup.
  • queue_timeout_seconds - default 300. After this delay an idle queue is dropped.

Step 5 - hosts/DNS rewrite (one-time)

The client only contacts the backend if the Epic domains are rewritten. The launcher DLL (RebootLauncher, NeoniteV2, ...) handles this for you. If you run a custom client, edit %WINDIR%\System32\drivers\etc\hosts to point *.ol.epicgames.com and account-public-service-prod03.ol.epicgames.com to your Neodyme host. Restart Fortnite.

Quick checklist when matchmaking does not start

Symptom in the lobbyWhat to check in Neodyme logs
"Connecting..." forever, no log line on the backend The launcher / hosts file is not rewriting matchmaking traffic to Neodyme. Check the launcher's redirect rules.
No available server for <gamemode> in <region> No entry in game-servers.json matches that gamemode with status: "online" in the requested region OR in any fallback region. Add or fix the entry and restart.
Server is picked but the player joins and sees random cosmetics The host account does not have the server role. Look for 403 on dedicated_server/QueryProfile in the logs. Run /account role <host> server.
Invalid bucketId format or Unknown playlist The playlist sent by the client is not declared in game_modes. Add it (use any of the existing entries as template) and restart.
Server is picked but the client never joins The ip + port in the entry must be reachable from the client machine (not just from the Neodyme host). On a single PC, 127.0.0.1 is fine; on LAN, use the host LAN IP. Open the UDP port in the firewall.
Same server keeps being assigned and locks out further matches Recycled every game_modes[mode].time_between_games minutes (default 25). Use /diagnostic or the dev panel to see assignment state, or lower the value for testing.
The matchmaker does not currently health-check your dedicated server. The status field is honoured as-is - if you set "status": "online" on a server that is actually down, players will be routed to it and get a connection error. Keep that field in sync manually for now.

Defaults shipped in config/game-servers.json

The repository ships with placeholder entries for NAW / NAE / EU / ASIA / OCE / BR / ME on 127.0.0.1:777x. These are not real production servers - they exist so the file is well-formed out of the box. Replace them with your own server addresses before opening matchmaking to players.
