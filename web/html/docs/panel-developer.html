Developer Panel

The Developer Panel is the technical control surface of the backend. It exposes 11 specialized tabs covering live configuration, file editing, content editing, asset management, the shop editor, statistics, console logs, plugin management, command shortcuts, system health checks and a real-time WebSocket monitor.

Access requires the developer or owner role. All writes are CSRF-protected and audit-logged.

Quick navigation — Overview · Configuration · Config Files · Content Editor · Assets · Shop Editor · Stats · Logs · Plugins · Commands · System · Live Monitor

1. Overview

Landing tab with at-a-glance KPIs:

  • Server Status, Uptime, Heap Memory, Version (build number from server.properties)
  • Quick Actions: Force Shop Rotation, Refresh Status, Toggle Maintenance

Maintenance mode flips maintenanceMode server-side. While on, the lightswitch endpoint returns DOWN and players get the maintenance message you configured.

4. Content Editor NEW

Typed editors for 17 in-game content sections — no more raw JSON editing for MOTD, news, login messages, BR/Creative/STW content. Each section has a schema-driven form with locale switching, list reordering, duplication, deletion, and an upstream-reset button.

Layout

A left sidebar groups every editable section by domain:

  • Global: Login Message, Emergency Notice v1/v2, Lobby, Special Offer Video, Subgame Select, Subgame Info
  • Battle Royale: MOTD modern, BR News (legacy), BR News v2 (legacy), Battle Pass About
  • Creative: Creative News, Creative Ads, Creative Features
  • Save the World: STW News, Athena Message, Survival Message
  • Danger zone: Reset all content (re-downloads both motd.json and content-pages.json from the GitHub repo)

The 13-locale switcher

Above every editor, a row of 13 locale buttons (EN / AR / DE / ES / ES-419 / FR / IT / JA / KO / PL / PT-BR / RU / TR) controls what locale i18n fields display. Clicking another locale instantly re-renders every i18n field at that locale. Edits are scoped to the current locale.

i18n field toggle

Each translatable field has a checkbox labeled i18n:

  • OFF — the field is a plain string (useful for things like title: "v1.2.6"). The "plain string" badge shows in grey.
  • ON — the field is a 13-locale object. A coverage badge shows next to the label: 13/13 translations (green), 7/13 (orange), 0/13 (red).

Toggling preserves what you had: switching ON converts the string to { en: "..." }, switching OFF takes the current locale's value back as a plain string.

List sections (cards)

Sections like MOTD modern, BR News, Battle Pass About show their entries as collapsible cards. Each card has Move up / Move down / Duplicate / Delete in the header. Duplicating a MOTD modern entry auto-regenerates its contentId, tcId and contentHash to avoid collisions.

Image fields

Each image URL field includes:

  • The URL input itself
  • An Upload button (file picker → uploads via the Assets pipeline → URL auto-filled)
  • A Browse button (opens the image picker modal with search and folder grouping)
  • A live thumbnail at the right; broken URLs show "image not loading" with the full URL in a tooltip

Unsaved-changes safety

  • A pulsing orange dot appears on the Save button as soon as you change anything
  • Switching to another section asks for confirmation if you have unsaved edits
  • Closing the browser tab triggers the native "leave site?" prompt

Reset

  • Reset file (red button in the editor header) — re-downloads only the file behind the current editor from https://raw.githubusercontent.com/Aorux01/Neodyme/refs/heads/main/…. Confirmation required.
  • Reset all content (danger zone in sidebar) — re-downloads both motd.json and content-pages.json at once. Local edits are lost.

Validation

  • MOTD modern: contentId is required and must be unique across entries. Duplicates are refused at save time with the row numbers pointed out.
  • Locale codes are normalized; case mismatches (pt-br vs pt-BR) are caught.

5. Assets NEW

Manage every image served at /images/*. Three sub-views:

Upload

  • Drag-and-drop or click the dropzone to pick a file
  • Allowed types: PNG, JPEG, WebP, GIF — max 5 MB
  • Validation is done with magic-byte sniffing: a renamed .js file claiming to be a PNG is rejected
  • The file is written to public/images/uploaded-images/<timestamp>-<rand>.<ext>
  • It is registered in content/assets-index.json with metadata: originalName, uploadedBy, uploadedAt, size, tags: ["uploaded"]
  • The folder images/uploaded-images/** is added to alwaysLocal automatically, so the file is always served from disk — never redirected to a CDN
  • Result card includes the public URL and a one-click Copy button
  • "Recently uploaded" grid at the bottom shows the last 12 uploads

Browse local

Recursive scan of public/images/, grouped by top-level folder (uploaded-images/, neodyme-public-service/, radios/, …). For every file:

  • Thumbnail (lazy-loaded)
  • Filename, size, extension
  • Copy URL — copies the public URL to clipboard
  • Delete — only on uploaded-images/; staff-installed assets cannot be deleted from the panel

Browse online

Lists every entry in assets-index.json that has a CDN URL and is not present on disk — these are served by 302 redirect in online mode. Useful to know what the server could serve locally if you switch to local mode.

6. Shop Editor BETA

Hand-pick what fills each slot of data/shop.json. The slot layout (6 Daily + 8 Featured by default) comes from config/shop.json → shopCategories[].count.

Per-slot actions

  • Pick — opens a searchable cosmetic picker backed by fortnite-api.com/v2/cosmetics/br. Filter by name or CID, rarity-coloured borders. Pick → suggested price by rarity (legendary 2000, epic 1500, rare 1200, uncommon 800, common 200) → confirm
  • Random — rerolls just this slot from the eligible pool (rarity filter + shopExcludedItems)
  • Edit — manually tweak the grant string and price
  • Clear — remove the entry; next auto-rotation will refill

Bulk action

Randomize all empty rerolls every unset slot in one go.

Important caveat

The auto shop rotation (autoShopRotation) will overwrite manual picks at the next rotation. To keep a fixed shop, disable auto rotation in Configuration → Features.

7. Statistics

Detailed runtime stats:

  • Node.js version, platform, architecture, CPU count
  • Memory: heap used / heap total / RSS / external
  • System memory: total / free
  • Database type (JSON or MongoDB), accounts count, file sizes on disk
  • SSL status: protocol, certificate path & modification date

8. Logs

Streaming view of the server's console output. Filter by level (info / warn / error / debug). The auto-refresh toggle re-fetches every second; switching tabs auto-pauses it.

If debugRequests or debugResponses is on (Configuration → Debug), expect a flood — turn them off when not actively debugging.

9. Plugins

The plugin manager. List installed plugins, browse the remote store, install / update / uninstall by version, edit per-plugin config files (each plugin can ship its own JSON configs under plugins/<name>/*.json).

See the dedicated Plugins documentation for the manifest format and dependency rules.

10. Commands

Click-to-run shortcuts for the most frequent dev console commands, grouped by category:

  • Server: reload, reload config, reload plugins, reload shop, diagnostic
  • Data: backup now, migrate JSON↔Mongo (with dry-run)
  • Shop: rotate now, featured override
  • Players: lookup, sessions, give item / V-Bucks / XP

The output is streamed back into the panel so you don't need a terminal session.

11. System

Live health checks of the 8 critical subsystems — HTTP server, database, XMPP, matchmaking, shop manager, plugin system, Redis (if enabled), auth service. Each check returns a status (ok / disabled / error) and a message.

The chart at the bottom shows historic system-test pass rate over the last 24 hours.

12. Live Monitor

Real-time charts for CPU, RAM, request rate, and a live list of connected XMPP clients. Polls every 2 seconds; auto-stops when you switch tabs to save bandwidth.

Safety & audit

Every write through the Dev Panel goes through CSRF protection (cookie + header) and is recorded in the audit log with the new update_content, reset_content, upload_asset, delete_asset, update_shop_slot, clear_shop_slot, randomize_shop_slot action types. The Admin Panel's Audit Log shows the full before/after diff for each change.
