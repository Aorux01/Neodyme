Moderation Panel

The Moderation Panel is the day-to-day workspace for the support team. It handles support tickets, in-game player reports, bans, account lookups, and creator-code approvals. Access requires the moderator, admin, or owner role.

Heads up — The Moderation, Developer and Admin panels are all hosted on a single page (web/html/admin.html) reached via the dashboard's "Staff Panel" link. The page only renders the sections your role is authorized to see; non-staff are redirected to the regular dashboard.

Accessing the panel

  1. Sign in on the dashboard at https://your-server.tld/
  2. If your role allows it, a "Staff Panel" link appears in the dashboard header
  3. The page route is /admin.html — bookmarkable
  4. The page checks your role via GET /neodyme/api/user/role on load; if you are not staff, you are redirected back to the dashboard

1. Tickets

Players open tickets from their dashboard ("Help & Support"). Each ticket carries a category, a message, an optional attachment, and a status (open, in_progress, closed).

Typical workflow

  1. A new ticket arrives in Open — it's highlighted
  2. You can assign yourself (POST /tickets/:id/assign) so other staff see who owns it
  3. Add internal notes — only staff see them
  4. Send replies — the player gets a dashboard notification
  5. Close when resolved (POST /tickets/:id/close) or delete for spam (DELETE /tickets/:id, admin-only)

Audit trail

Every ticket action you take is recorded in the Audit Log with the ticket ID and the action (assign_ticket, close_ticket, delete_ticket).

2. Player Reports

When a player uses Fortnite's in-game report UI (right-click on a player, "Report"), the data is forwarded to the Reports tab. Each entry shows the reporter, the target, the category (cheating, toxicity, exploit…), and the in-game context where it happened.

Actions per report

  • Open a ticket from the report — opens a pre-filled ticket targeted at the reported player, useful if a real investigation is needed
  • Quick ban — opens the ban modal with the target pre-filled
  • Dismiss — mark as not actionable; the entry stays in the audit log
  • Dismiss all — bulk-clear if a wave of false reports comes in

Note on duplicates

Reports are not deduplicated server-side — if 5 players report the same target you'll see 5 entries. The reporter column lets you filter and triage quickly.

3. Player Search & Ban

The Players tab lets you search a player by display name, email or account ID and see:

  • Account creation date, last login, last IP
  • Current role (player / founder / helper / moderator / developer / admin / owner / server)
  • Current ban status — if banned, reason, duration and the moderator who banned them
  • Active sessions (live access tokens; revoking them logs the player out everywhere)

Ban modal

Triggered via the Ban button on a player row or from a report. Fields:

FieldRequiredNotes
ReasonYesStored verbatim in the audit log and shown to the player on next login
DurationYesPick a preset (1h / 1d / 7d / 30d / Permanent) or a custom duration. Permanent never auto-expires.
Invalidate sessionsNo (default ON)Force-revokes every active token so the player can't finish their current match

Unban

Click Unban on a banned player. You'll be asked for a justification (also logged). Players are not notified that they were unbanned; they just succeed at their next login.

4. Creator Codes

Players can request a SAC code from their dashboard. Requests appear in this tab.

Per-request actions

  • Approve — activates the code; the player now earns the configured commission (creatorCodeCommissionPercent) on every shop purchase that uses it
  • Reject — refused with no commission. The player can request again with a new code string after.
  • Toggle — temporarily disable an already-approved code (e.g. ToS violation under investigation)
  • Delete — remove permanently. The player can then request a new code.

Naming rules

Codes are case-insensitive, A–Z and 0–9 only, 3–16 characters. Profanity filtering is not built-in — staff must review manually before approving.

5. Feedback

Players can submit free-form feedback (one-shot, not a ticket) via their dashboard. This tab is a flat list with timestamp, account ID and content.

  • Mark as read — visual only, doesn't remove the entry
  • Delete — permanently remove. Logged in audit.

Audit log

All moderation actions are recorded server-side. Moderators see their own actions in the Admin Panel's Audit Log; admins see everyone's. Each row contains:

  • Action type (ban_user, unban_user, close_ticket, approve_code…)
  • Who performed it (display name + account ID + IP)
  • Target (user, ticket, code…)
  • Detailed payload: ban reason, ticket message, before/after values, etc.

Common pitfalls

  • Don't change your own role — the API will refuse, but if you somehow end up role-less you lose panel access
  • Permanent bans without a clear reason — the reason field is shown to the player and stored forever; write it as if it were public
  • Quick-banning from a report without checking sessions — the player can finish the match if you forget to invalidate sessions
  • Deleting a ticket — loses the message history. Prefer close unless it's outright spam.
