Plugin System

Neodyme features a plugin system that allows you to extend server functionality without modifying the core codebase. Plugins are JavaScript modules loaded dynamically at server startup.

Configuration

The plugin system is controlled via server.properties:

plugins=true    # Enable/disable plugin loading

When plugins=true, the server automatically scans the plugins/ directory at startup and loads all valid .js files or subdirectories containing an index.js file.

Plugin Structure

Minimal Template

class MyPlugin {
    constructor() {
        this.name = "MyPlugin";
        this.version = "1.0.0";
        this.description = "My plugin description";
        this.author = "Your Name";
        this.license = "MIT";
        this.repository = "https://github.com/username/my-plugin";
        this.minBackendVersion = "1.2.0";

        this.dependencies = {
            npm: ["axios", "moment"],           // NPM packages
            plugins: ["other-plugin-name"]      // Required plugins
        };
    }

    async init(pluginManager) {
        // Initialization code
        return true; // return false to abort loading
    }

    async shutdown() {
        // Resource cleanup
    }
}

module.exports = MyPlugin;

Validation Rules

The PluginManager checks these conditions before loading:

  1. plugin.name must be defined and non-empty
  2. plugin.init must be a function
  3. If minBackendVersion is defined, current backend version must be >= that value

Examples

Basic Plugin

const LoggerService = require('../src/service/logger/logger-service');

class HelloPlugin {
    name = "HelloPlugin";
    version = "1.0.0";
    description = "Demo plugin";
    author = "Your Name";
    minBackendVersion = "1.2.0";

    async init(pluginManager) {
        LoggerService.log('success', 'HelloPlugin initialized');
        return true;
    }

    async shutdown() {
        LoggerService.log('info', 'HelloPlugin stopped');
    }
}

module.exports = HelloPlugin;

Plugin with Periodic Tasks

class SchedulerPlugin {
    name = "SchedulerPlugin";
    version = "1.0.0";

    constructor() {
        this.intervals = [];
    }

    async init(pluginManager) {
        const interval = setInterval(() => {
            this.periodicTask();
        }, 5 * 60 * 1000); // every 5 minutes

        this.intervals.push(interval);
        return true;
    }

    periodicTask() {
        LoggerService.log('info', 'Periodic task executed');
    }

    async shutdown() {
        // Always clean up intervals
        this.intervals.forEach(i => clearInterval(i));
        this.intervals = [];
    }
}

module.exports = SchedulerPlugin;

Plugin Store

Browse, install, and update plugins directly from the console:

/plugins store list                                # Browse available plugins
/plugins store search discord                      # Search by keyword
/plugins store info discord-integration            # Show details + version matrix
/plugins store install discord-integration         # Install latest compatible version
/plugins store install discord-integration 1.0.0   # Install a specific version
/plugins store update discord-integration          # Update to latest compatible
/plugins store update discord-integration 1.0.1    # Update to a specific version
/plugins store uninstall discord-integration       # Uninstall plugin
/plugins store refresh                             # Force-refresh the store cache

Without an explicit version, the installer picks the highest version whose minBackendVersion is satisfied by the running backend. If you ask for a version that does not exist or that requires a newer backend, the installer aborts and prints the list of available versions.

Download Progress

Multi-file downloads (plugin install, future image bundles, etc.) render a single global progress bar rather than one bar per file. The total size is resolved up front from the manifest (size per file or totalSize per version) or via parallel HEAD probes, and the bar grows gracefully if the total is unknown.

Manifest Format

A plugin manifest is the JSON file referenced by manifestUrl in the global store list. Two layouts are accepted; the grouped versions object is recommended, the flat files[] layout is kept for backwards compatibility.

Recommended: grouped versions

File Size Resolution

The progress bar uses the first available source from this priority list:

  1. file.size per entry (most precise)
  2. versions[v].totalSize for the resolved version
  3. manifest.totalSize at the root
  4. Parallel HEAD probes on every URL
  5. Unknown - the bar grows live with received bytes

Dependencies

NPM Dependencies

Listed under dependencies.npm. When you install a plugin from the store, the installer runs npm install --save <packages> for you automatically — you do not need to install anything by hand. For manually placed plugins (drop-in .js files), install dependencies yourself:

npm install discord.js axios moment

Plugin-to-Plugin Dependencies

Listed under dependencies.plugins. Each entry is either a plain plugin id string or an object with an explicit target version:

"dependencies": {
  "npm": ["discord.js"],
  "plugins": [
    "logger-utils",
    { "id": "shop-tools", "version": "2.1.0" }
  ]
}

Before downloading the requested plugin, the installer resolves every plugin dependency recursively:

  • If the dependency is already installed, it is skipped (no version downgrade or replacement).
  • Otherwise, installPlugin() is called for it, with the requested version (or latest compatible if none).
  • If any dependency fails to install, the whole install is aborted with an explicit error.

Cycle Detection

The install chain is tracked through a trail: Set<string> shared across recursion. If a plugin re-appears in its own dependency chain (A → B → A), the second visit is short-circuited with a warning instead of looping forever:

[warn] Skipping 'plugin-a' (already in install chain, possible dependency cycle)

This keeps installs deterministic even with badly authored manifests, and lets you safely express bidirectional "these two plugins know about each other" relationships.

Best Practices

  • Always wrap init() in a try-catch and return false on failure
  • Implement shutdown() if you use intervals, connections, or event listeners
  • Use path.join(__dirname, ...) for relative file paths
  • Set minBackendVersion to the oldest compatible version
  • For store plugins, always include license, repository, and bugs metadata
